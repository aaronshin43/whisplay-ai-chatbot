## 1. 발견된 문제: LLM 응답 왜곡 (치명적 안전 이슈)

### 발생 경위

v1 최적화(TTFT 개선, KV 캐시 워밍업, 배치 테스트) 이후, LLM이 프로토콜에 충실하도록 
시스템 프롬프트를 강화했다:

```
RULES:
- Your response MUST come ONLY from the ACTIVE PROTOCOL below.
- NEVER add advice beyond what the protocol states.
- If the protocol says "do not", you MUST say "do not".
```

**결과: 1.7B 모델이 핵심 지시사항을 정반대로 뒤집기 시작했다.**

### 관찰된 위험 응답

| 질문 | 프로토콜 원문 | LLM 실제 응답 |
|------|-------------|--------------|
| "아이가 뭔가 삼키고 숨을 못 쉬어" | 등 두드리기 5회 + 가슴 압박 5회 | "등 두드리기, 가슴 압박, 복부 밀어올리기를 하지 마세요" |
| "쓰러져서 숨을 안 쉬어" | CPR 시작 — 세게 빠르게 압박 | "하지 마세요." |
| "가슴 통증에 땀 흘리고 있어" | 아스피린(325mg) 씹어 먹이기 | "아스피린을 주지 마세요" |
| "떨림이 멈췄어" | 코어 부위를 따뜻하게 | "젖은 옷을 벗기지 마세요" |

### 근본 원인

1.7B 모델이 시스템 프롬프트의 "하지 마라" 규칙에 과도하게 반응했다. 프로토콜 안에 있는 
긍정적 행동 지시("~해라")에까지 부정을 적용하여 의미를 뒤집었다. 모델이 메타 규칙("프로토콜 
밖의 조언을 하지 마라")과 프로토콜 내부 지시("~해라")를 구분할 추론 능력이 부족하다.

**이것은 프롬프트 엔지니어링의 문제가 아니라, 1.7B 파라미터 모델의 근본적 한계다.**

## 2. 해결: Direct Protocol Delivery (LLM 바이패스)

### 핵심 원칙

첫 응답에서 LLM을 완전히 제거한다. 사람이 작성하고 검토한 프로토콜 텍스트를 
LLM 해석 없이 사용자에게 직접 전달한다.

### 아키텍처 변경

```
변경 전 (v1 — 모든 응답이 LLM 경유):
  사용자 질문 → 임베딩 매칭 → 프로토콜 발견 → LLM 해석 → 응답
  문제: LLM이 지시사항을 왜곡/반전

변경 후 (v2 — 직접 전달):
  사용자 질문 → 임베딩 매칭 → 프로토콜 발견 → 프로토콜 텍스트 직접 스트리밍
  LLM 미개입. 왜곡 가능성 제로.
```

### LLM 사용 범위 (v2)

| 상황 | 처리 방식 | LLM 사용? |
|------|----------|----------|
| 첫 질문, 프로토콜 매칭 성공 | 프로토콜 텍스트 직접 전달 | 아니오 |
| 같은 프로토콜 후속 질문 | 간소화된 프롬프트로 LLM 응답 | 예 |
| 매칭 실패 (TRIAGE) | LLM이 상황 파악 질문 | 예 |

LLM은 오직 두 가지 경우에만 사용: (1) 이미 전달된 프로토콜에 대한 후속 Q&A, 
(2) 상황이 불분명할 때 "무슨 일이 일어났나요?" 질문. 둘 다 핵심 응급처치 
지시를 처음 생성하는 역할이 아니므로 왜곡 위험이 크게 감소.

### 3.2 OasisAdapter 반환값 확장

**파일:** `src/core/OasisAdapter.ts`

새 `OasisResult` 인터페이스 — 프로토콜 원문과 메타데이터를 함께 반환:

```typescript
interface OasisResult {
  protocolId: string;    // 예: "SNAKEBITE"
  protocolText: string;  // 직접 전달용 프로토콜 원문
  systemPrompt: string;  // 후속 질문 LLM용
  isTriage: boolean;     // 매칭 실패 시 true
  score: number;         // 임베딩 유사도 점수
}
```

새 함수 `matchOasisProtocol()`이 기존 `getSystemPromptFromOasis()`를 대체.

후속 질문 프롬프트 (간소화):
```
당신은 응급 보조원입니다. 아래 프로토콜만 사용하여 후속 질문에 답하세요.
프로토콜의 관련 문장을 직접 인용하세요. 짧게 답하세요.
```

TRIAGE 프롬프트:
```
당신은 응급 보조원입니다. 상황이 불분명합니다.
무슨 일이 일어났는지 파악하기 위한 질문을 하나만 하세요.
```

### 3.3 ChatFlow — 프로토콜 직접 스트리밍

**파일:** `src/core/ChatFlow.ts`

프로토콜 추적 상태 추가 (`currentOasisProtocolId`, `currentOasisResult`)와 조건 분기:

- **프로토콜 매칭 + 첫 질문** → 프로토콜 텍스트를 청크 단위로 직접 스트리밍 (LLM 호출 없음)
- **같은 프로토콜 + 후속 질문** → 간소화된 프롬프트로 LLM 응답
- **TRIAGE** → LLM이 상황 파악 질문

## 6. 알려진 한계 및 향후 고려사항

### 현재 한계

1. **후속 질문은 여전히 LLM 사용** — 간소화된 프롬프트로 왜곡 위험을 줄였지만, 
   후속 Q&A에서 의미 반전 가능성이 완전히 사라지지는 않았다.

2. **프로토콜 텍스트 길이** — 현재 프로토콜은 3~5문장(150~280자). 프로토콜이 
   길어지면 전체를 한 번에 전달하는 것이 비효율적일 수 있다.

3. **쿼리 맥락 미반영** — "독 빨아내도 돼?"라고 물어도 프로토콜 첫 문장부터 
   전달한다. 가장 관련 있는 문장을 먼저 전달하지 않는다.

### 검토했으나 미구현

| 방법 | 설명 | 미채택 이유 |
|------|------|-----------|
| Step 단위 분리 | 프로토콜을 번호 매긴 단계로 쪼개서 하나씩 전달 | 사용자 질문 맥락 무시, "다음" 인터랙션 패턴 필요 |
| 하이브리드 (첫 문장 직접 + 나머지 LLM) | 첫 문장만 직접 전달, 나머지는 LLM | 나머지 콘텐츠에서 LLM 왜곡 재발 |
| 쿼리 인식 임베딩 | 프로토콜 문장별 별도 임베딩 | Pi에서 추가 메모리/연산 부담, 임베딩 ≠ Q&A 관련성 |
| Smart Sentence Reordering | 키워드 오버랩으로 관련 문장 우선 배치 | 현재 프로토콜 길이(3~5문장)에서는 복잡도 대비 효과 미미 |
| LLM 완전 제거 | 후속 질문에서도 LLM 미사용, 프로토콜/FAQ만 | 유연한 후속 질문 대응 불가 |
| 위험도별 LLM 접근 제어 | 생명 위협 프로토콜은 LLM 차단, 저위험은 허용 | 구현 가치 있으나 프로토콜별 위험도 분류 필요 |
| 프로토콜별 FAQ | 예상 후속 Q&A 쌍을 키워드로 매칭 | 장기 전략으로 유효, 의료 검수 필요 |

### 근본적 트레이드오프: 안전성 vs 유연성

```
안전성 ◀──────────────────────▶ 유연성
(프로토콜만)                    (LLM 자유응답)

"독 빨아내지 마라"              "독을 빨아내세요"
← 정확하지만 딱딱함              ← 유연하지만 치명적으로 틀림
```

1.7B 모델의 한계로 인해, 유연한 응답과 안전한 응답을 동시에 달성할 수 없다.
프로토콜에 없는 추론이 필요한 질문(예: "티셔츠 써도 돼?", "진통제 줘도 돼?", 
"병원까지 3시간인데 운전해야 해?")에는 LLM 없이 답할 수 없다. 
하지만 LLM에게 의료 추론을 맡기면 의미 반전의 위험이 있다.

### 권장 다음 단계 (우선순위)

1. **프로토콜에 `severity` 필드 추가** — 각 프로토콜을 "생명 위협", "긴급", 
   "예방"으로 분류. 후속 질문에서 LLM 접근을 위험도에 따라 제어.

2. **Smart Sentence Reordering** — 특정 행동에 대한 질문("독 빨아내도 돼?")일 때, 
   프로토콜 문장 중 관련 키워드가 포함된 문장을 맨 앞으로 재배치. 
   단순 키워드 오버랩으로 구현, ML 불필요.

3. **FAQ 확장** — 생명 위협 프로토콜(CPR, CHOKING, ANAPHYLAXIS 등)에 
   사전 작성된 후속 답변을 추가하여 LLM 의존을 완전히 제거.

4. **하드웨어 업그레이드 평가** — NPU 탑재 보드(RK3588) 또는 Jetson Orin Nano에서 
   3B~7B 모델 테스트. 더 큰 모델이면 왜곡 문제 자체가 해결될 수 있다.

## 8. 핵심 결론

> 1.7B 파라미터 모델은 제한된 하드웨어에서 응급 의료 프로토콜을 안전하게 해석할 수 없다.
> 모델이 작을수록 핵심 지시사항을 반전시킬 가능성이 높아진다.
> **생명이 걸린 응용에서는 LLM을 우회하고 사람이 작성한 프로토콜 텍스트를 직접 전달하는 것이
> 제약이 아니라, 올바른 아키텍처다.**
